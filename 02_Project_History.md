# 02_Project_History.md

## Purpose

本文件用于记录 User 的项目经历。

它不是简历，也不是对外提交用文档，而是 Career Branding 项目的项目素材库。  
后续制作以下内容时，应优先参考本文件：

- 自己紹介
- 派遣先初日挨拶
- 面试回答
- 自己PR
- 技术说明
- STAR 故事
- 项目经验整理
- 职业成长总结

本文件的重点不是简单记录“做过什么”，而是整理每个项目中体现出的能力、思考方式、困难、解决过程和未来可复用价值。

---

## Status Labels

本文使用以下标记：

- `[Confirmed]`：User 已明确确认的信息
- `[Inference]`：根据目前信息和常见研修 / 项目情况进行的合理推测
- `[Need Confirmation]`：需要 User 后续确认的信息
- `[Draft]`：暂定表达，后续可调整
- `[Deprecated]`：已不建议继续使用或已过时的信息

---

## How to Use This File

Assistant 在处理项目经历时，应优先参考本文件。

每个项目应尽量整理为以下结构：

1. 项目概要
2. 使用技术
3. User 负责内容
4. 遇到的问题
5. 解决过程
6. 学到的内容
7. 可用于自己紹介 / 面试的表达
8. STAR 化可能性
9. 待确认事项

本文件应随着 User 的研修、派遣、项目经验持续更新。

# 10. Risk Management

## Risk 1 — Project Details Are Still Vague

多くのプロジェクトについて、User の具体的な担当範囲が未確認。

Countermeasure:

- 自己紹介では広めに表現する
- 面接では具体的な担当範囲を確認してから使う
- `[Need Confirmation]` を残す

## Risk 2 — Overclaiming

研修課題を実務経験のように表現すると過剰に聞こえる。

Avoid:

- 実務で開発しました
- 主導しました
- 高度な実装を行いました
- 豊富な開発経験があります

Prefer:

- 研修課題で取り組みました
- 基礎を学びました
- 実装や動作確認を経験しました
- 確認しながら進めました

## Risk 3 — Too Many Projects in Self-Introduction

自己紹介で全てのプロジェクトを説明すると散らかる。

Countermeasure:

- 派遣先自己紹介では 2～3 個までに絞る
- 代表例として Timer と音楽プレーヤーを使う
- 他のプロジェクトは「グループ課題として〜など」とまとめる

## Risk 4 — Follow-up Questions

具体的に話しすぎると、細かい技術質問を受ける可能性がある。

Countermeasure:

- 自信を持って説明できる範囲に限定する
- 不確かな部分は抽象度を上げる
- 面接前に STAR 化して準備する
